# RLM Tool: Parallel Document Analysis for the Agent

## Problem

The agent can search the library and read individual documents, but it cannot efficiently analyze content across many documents at once.

### Current flow (without RLM)

User asks: "Compare the methodology across my ML papers"

```
Agent loop iteration 1: search_semantic("ML methodology") → 15 results
Agent loop iteration 2: read_library_item(paper_1) → full text
Agent loop iteration 3: read_library_item(paper_2) → full text
...
Agent loop iteration 16: read_library_item(paper_15) → full text
Agent loop iteration 17: (finally synthesize, if context window not exhausted)
```

Problems:
- **15+ round trips** to the LLM just for reading (each is a full API call with the growing conversation context)
- **Context window fills up** — 15 papers at ~5K tokens each = 75K tokens of raw text crammed into the conversation
- **Expensive** — the main model (Opus/Sonnet) processes all that text at full price
- **Slow** — sequential reading, one document per iteration
- **Hits max iterations** — the agent loop caps at 10 iterations, so it cannot even read 15 documents

### With RLM

```
Agent loop iteration 1: search_semantic("ML methodology") → 15 results
Agent loop iteration 2: rlm_analyze(item_ids, map_prompt, reduce_prompt) → synthesis
Agent loop iteration 3: (agent responds with the comparison)
```

What happens inside that single `rlm_analyze` call:
1. Fetch all 15 documents' text from the database (already chunked and stored)
2. Send 15 parallel requests to a cheap model (e.g., Haiku at $0.25/1M tokens), each extracting methodology details
3. Collect the 15 short summaries
4. Send one request to the main model to synthesize into a comparison
5. Return the synthesis as a single tool result

## Why it matters

### Cost

| Approach | Main model tokens | Cheap model tokens | Estimated cost |
|----------|------------------|--------------------|---------------|
| Sequential reads | ~150K (15 docs in context) | 0 | ~$2.25 (Sonnet) |
| RLM | ~10K (synthesis only) | ~75K (15 x 5K) | ~$0.17 |

13x cheaper for a 15-document analysis.

### Speed

| Approach | API calls | Parallelism |
|----------|-----------|-------------|
| Sequential reads | 17+ (serial) | None |
| RLM | 1 reduce + 15 map (10 concurrent) | 10x parallel |

### Context window

| Approach | Context consumed |
|----------|-----------------|
| Sequential reads | ~150K tokens (raw text accumulates in conversation) |
| RLM | ~10K tokens (only the synthesis enters the conversation) |

The agent's conversation stays clean. It sees the synthesis, not 15 raw documents.

## Use cases

### Common: Post-retrieval analysis (10-50 docs)

Semantic search narrows 1K documents to the top 20 relevant ones. RLM analyzes those 20 in parallel. This is the default path.

- "What do my papers say about attention mechanisms?"
- "Find contradictions across my saved articles about climate policy"
- "Extract all datasets mentioned in my NLP papers"
- "Which papers cite the same foundational work?"

### Rare: Full-library scan (up to 50 docs per call)

When vector similarity is insufficient — the information is too specific or structural for embeddings to capture.

- "Which of my papers have experiments on ImageNet?" (embedding won't reliably match a dataset name in a results table)
- "Find all papers where the first author is affiliated with MIT" (structural metadata, not semantic)
- "Tag each paper with its primary research method: survey, experiment, theoretical"

### Batch extraction

- "Extract the abstract and key findings from each of my unread papers"
- "Generate a one-sentence summary for every paper in my 'To Review' collection"

## Design

### Agent interface

The agent calls it as a regular tool with dynamic prompts:

```json
{
  "name": "rlm_analyze",
  "input": {
    "query": "transformer architectures",
    "map_prompt": "Given this paper:\n\nTitle: {title}\n\n{document_text}\n\nExtract: 1) The specific transformer variant proposed 2) Key architectural innovation 3) Reported performance metrics",
    "reduce_prompt": "You are given methodology extractions from multiple papers. Compare them and identify common themes, unique approaches, and contradictions.",
    "max_documents": "20"
  }
}
```

The agent writes the prompts dynamically based on the user's question. This is what makes it an RLM tool rather than a fixed "batch analysis" function — the intelligence is in the prompts, not the code.

### Internal architecture

```
rlm_analyze tool call
│
├─ 1. Resolve documents
│     query → SemanticIndexService.search() → item IDs
│     OR parse explicit item_ids
│
├─ 2. Fetch text
│     SQL: semantic_chunks WHERE item_id IN (...) ORDER BY page_start
│     Concatenate chunks per document (already stored from indexing)
│
├─ 3. Map phase (Swift TaskGroup, 10 concurrent)
│     For each document:
│       Substitute {title}, {document_text} into map_prompt
│       → cheap model (Haiku/Flash) → short extraction
│     Per-task error isolation (failures don't abort others)
│
├─ 4. Reduce phase
│     Collect all map results, label by document title
│     → main model with reduce_prompt as system prompt
│     → synthesis
│
└─ 5. Return ToolOutput.success(synthesis + metadata)
```

### Map model selection — use the cheapest available

The entire cost advantage of RLM depends on the map phase using a **genuinely cheap model**. If someone maps with Opus, it's actually more expensive than sequential reads. The design makes this hard to get wrong.

**The map phase can use a completely different provider than the main conversation.** For example: main chat is Anthropic Opus, map phase is DeepSeek Flash.

**Cheap model candidates already registered in OakReader:**

| Provider | Model | ~Input pricing | Notes |
|----------|-------|---------------|-------|
| DeepSeek | `deepseek-v4-flash` | $0.07/1M tokens | Best value, good Chinese support |
| Google | `gemini-2.5-flash-lite` | Free / $0.015/1M | Free tier available, 1M context |
| ZAI (GLM) | `glm-4.5-air` | ~$0.04/1M | Good Chinese support |
| Kimi | `kimi-k2-turbo-preview` | ~$0.06/1M | Good Chinese support |
| Groq | `llama-3.1-8b-instant` | Free tier | Fast but small model |
| Mistral | `ministral-3b-latest` | ~$0.04/1M | 3B model, limited quality |
| OpenAI | `gpt-4.1-nano` | ~$0.10/1M | Cheapest OpenAI |
| Anthropic | `claude-haiku-4-5` | $0.80/1M | 10x more expensive than DeepSeek |

**Cost comparison for 15-document analysis (~75K map tokens):**

| Map model | Map cost | + Reduce cost | Total |
|-----------|----------|---------------|-------|
| DeepSeek V4 Flash | $0.005 | $0.15 | **$0.16** |
| Gemini 2.5 Flash Lite | $0.001 | $0.15 | **$0.15** |
| GLM 4.5 Air | $0.003 | $0.15 | **$0.15** |
| Claude Haiku 4.5 | $0.060 | $0.15 | **$0.21** |
| Sequential reads (no RLM) | — | — | **$2.25** |

**Resolution order:**
1. **Explicit preference**: user sets `rlmMapProvider`/`rlmMapModel` in settings
2. **Auto-detect cheapest**: scan all providers where user has an API key (via `CredentialResolver.hasCredentials`), pick the cheapest model using a priority table
3. **Same-provider fallback**: if no other provider has credentials, use the cheapest model from the main provider

The auto-detect priority table (checked in order):
1. DeepSeek → `deepseek-v4-flash`
2. Google → `gemini-2.5-flash-lite`
3. ZAI → `glm-4.5-air`
4. Kimi → `kimi-k2-turbo-preview`
5. Groq → `llama-3.1-8b-instant`
6. Mistral → `ministral-3b-latest`
7. OpenAI → `gpt-4.1-nano`
8. Anthropic → `claude-haiku-4-5`

This means a user who has both an Anthropic key (for main chat) and a DeepSeek key (for cheap work) automatically gets the best of both worlds without any configuration.

## What it does NOT do

- Does not replace keyword or semantic search (those handle retrieval)
- Does not require Python or any external runtime (pure Swift async/await)
- Does not modify documents or the database (read-only)
- Does not add new dependencies or schema migrations
- Does not stream intermediate progress (returns final result as one tool output)
